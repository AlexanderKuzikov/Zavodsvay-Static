## Локальные шрифты

### Принцип именования
- Используются локальные имена семейств: `Open Sans Local` и `Roboto Slab Local`
- Суффикс `Local` нужен, чтобы исключить конфликт с системно установленными шрифтами и внешними совпадающими family-name
- В `body` используется `font-family: 'Open Sans Local', sans-serif;`
- В заголовках (`h1/h2/h3`, `.object-page__title`) используется `font-family: 'Roboto Slab Local', serif;`

### Правильное подключение кириллицы
- Для сабсетов с кириллицей и латиницей нельзя полагаться на «первый файл в src» без `unicode-range`
- Практический кейс проекта: кириллица была в `RobotoSlabCyr.woff2`, а цифры и знаки — только в `RobotoSlab.woff2`; без корректного разделения браузер ронял цифры в fallback
- Правильная схема: **два отдельных `@font-face` блока** на одно family, каждый со своим `unicode-range`
- Для `Roboto Slab Local` латинский файл покрывает `U+0000-00FF` и связанные латинские диапазоны; кириллический файл покрывает только кириллицу
- Для `Open Sans Local` используется та же схема раздельных сабсетов

### Проверка файлов шрифта
- Google Fonts может подсунуть не тот сабсет или не тот файл; имя файла не гарантирует правильное содержимое
- Перед интеграцией локальный `.woff2` нужно проверять через [wakamaifondue.com](https://wakamaifondue.com) — смотреть glyph coverage, unicode ranges и variable axes
- Если браузер показывает в DevTools `Rendered Fonts` fallback-семейство (например, `Georgia`/`serif`) при том что `font-family` CSS уже применился — проблема почти всегда в содержимом файла или покрытии глифов, а не в селекторе
- Отдельно проверять цифры, пунктуацию, знак №, латиницу и кириллицу — не только русские буквы

### Практическое правило
- Если шрифт разбит на `latin` и `cyrillic`, подключать их как отдельные `@font-face` записи с одним family-name и разными `unicode-range`
- Не переименовывать `.ttf/.otf` в `.woff2` вручную — это не конвертация
- Fallback у заголовков — просто `serif`, без `Georgia`: Georgia визуально слишком отличается от Roboto Slab

## Карта выполненных работ (реализовано)

### Стек
- Яндекс.Карты JS API v3, `@yandex/ymaps3-clusterer@0.0.1` via jsdelivr

### Данные
`data/map.json` — массив объектов:
```json
{
  "id": 3,
  "coords": [56.5178955, 57.9237226],
  "category": "house",
  "title": "...",
  "techDescription": "...",
  "images": ["3_1.webp"],
  "url": "/objects/3/"
}
```

> ⚠️ **Координаты в `data/map.json`** хранятся как `[latitude, longitude]` (бытовая нотация, как в интерфейсе Яндекс.Карт).  
> ymaps3 JS API также принимает их в формате `[latitude, longitude]` — **перевод НЕ нужен**, `obj.coords` подставляется напрямую.  
> Это подтверждено эмпирически: объекты на общей карте и на странице объекта отображаются точно, без сдвига.

### Ключевые решения и известные грабли ymaps3

| Проблема | Решение |
|---|---|
| Клик на маркер не работает | `onClick(event, mapEvent) { mapEvent.stopPropagation() }` в props маркера — не DOM-событие! |
| `stopPropagation` на DOM-событии не помогает | ymaps3 имеет собственную систему событий, отдельную от DOM |
| `__DIR__` в require-файле | Указывает на директорию шаблона, не вызывающего. Передавать `$object_dir = __DIR__` из index.php |
| `registerCdn` второй аргумент | Строка, не массив: `'@yandex/ymaps3-clusterer@0.0.1'` — явная версия обязательна |
| Порядок слоёв в дереве карты | Слои добавляются в порядке `addChild`. Маркеры/кластеры добавлять на `map`, а не на слой. Смена подложки — только первый слой, остальные не трогать |
| `behaviors` карты | `['drag', 'pinchZoom', 'scrollZoom', 'dblClick']` — все 4 обязательны |
| Координаты на странице объекта уезжают | Не делать swap `[lng, lat]`; использовать `obj.coords` напрямую и на общей карте, и в `_template.php` |

### Фильтрация по категориям (реализовано)

- Фильтр встроен в легенду под картой
- Клик по категории в состоянии «все включены» → оставляет только эту категорию (`solo-click`)
- Повторный клик по единственной активной категории → возвращает все категории
- В остальных случаях работает обычный toggle
- Есть кнопка `Показать все`, счётчик `показано N из M` — только при активном фильтре
- Реализация клиентская: `clusterer.update({ features })` без перезагрузки карты
- **Легенда идентична на `/map/` и на странице объекта** (HTML `.map-legend`, CSS, логика)
- На странице объекта: текущий объект всегда виден (отдельный маркер), в фильтрацию не включён; `otherObjects.filter(...)` передаётся в clusterer
- Счётчик на странице объекта: `показано N+1 из M` (текущий объект учитывается в числителе)

### Спутниковый слой и переключатель типов карты

> ⛔ **В ymaps3 (JS API v3) официального спутникового слоя нет.**  
> Яндекс подтвердил: спутник и гибрид в текущей публичной версии API 3.0 не поддерживаются.  
> Использование недокументированных `YMapTileDataSource` с тайлами `core-sat.maps.yandex.net` нарушает ToS и нестабильно (проверено — серый прямоугольник вместо карты).

**Варианты для заказчика (требует обсуждения):**
- Остаться на схеме ymaps3 — стабильно, без переключателя
- Переключение тем `light`/`dark` через `schemeLayer.update({ theme })` — возможно, не спутник
- Миграция на ymaps2 — полный переключатель схема/спутник/гибрид, но рефакторинг API
- Сторонние спутниковые тайлы (Mapbox, OSM) — отдельная лицензия

## Object Pages (реализовано)

### Архитектура
- `pages/objects/_template.php` — единый шаблон
- `pages/objects/{id}/index.php` — двустрочник:
```php
<?php
$object_id  = 3;
$object_dir = __DIR__;
require __DIR__ . '/../_template.php';
```
- Шаблон читает данные из `data/map.json` по `$object_id`
- Все пути через `$object_dir`, не `__DIR__` шаблона

### Карта на странице объекта
- Центрирование на объекте, zoom 13
- Текущий объект: отдельный маркер `zIndex: 100`, цвет `#f97316`, CSS-анимация `marker-pulse`
- Остальные объекты: кластеризация + навигация (те же что на `/map/`)
- Текущий объект исключён из clusterer (`.filter(o => o.id !== CURRENT_ID)`)
- Координаты используются напрямую из `data/map.json`, без swap
- **Легенда-фильтр** под картой объекта — идентична `/map/` по HTML (класс `.map-legend`), CSS и JS-логике

### Добавление нового объекта
1. Создать `pages/objects/{id}/index.php` (3 строки, см. выше)
2. Убедиться что объект в `data/map.json` с `"url": "/objects/{id}/"`
3. Добавить изображения: `pages/map/img/{id}_1.webp`, `{id}_2.webp`, ...
4. Обновить `sitemap.xml`

### SEO объекта
- Schema.org `CreativeWork` + `GeoCoordinates` в шаблоне
- `og_image` = первое изображение объекта
- `canonical` = `https://zavodsvay.ru/objects/{id}/`
